## 1. Canary api — état & ce qui reste

**Objectif** : valider que le chemin `navigateur → /api/data/* → Supabase` fonctionne
de bout en bout sur MON compte avant tout flip global.

Fait :
- Cookie `suimini_data_layer=api` posé (session courante).
- `window.__suiminiDataLayer()` → `"api"` ✅ → le bundle servi est bien le code
  runtime `getDataLayer` (plus l'ancien build-time ; SW en **v5**).
- `/api/data/whoami` → `authenticated: true` (résolution de session serveur OK,
  cookie + Bearer durcis dans `getServerAuth`).
- `/api/data/trees` → `200` avec données (pendant le diagnostic incident).

**À VALIDER (bloquant avant flip)** :
- [ ] **TEDA s'affiche-t-il en mode api ?** Onglet Réseau : les lectures doivent
      partir vers **`/api/data/trees`** (pas `*.supabase.co/rest/v1/*`), et l'arbre
      (71 membres) doit se rendre.
- [ ] Une **écriture** (ajout/édition d'une personne) passe par
      **`/api/data/trees/{id}/save`** et persiste (vérifier après reload).
- [ ] Une **suppression** → `/children/delete` (soft-delete) + pas de résurrection.
- [ ] Les **RPC** (ex. partage) → `/api/data/rpc/[name]` (16 RPC whitelistées).

**Rollback canary** = effacer le cookie `suimini_data_layer` (retour `direct`
immédiat, sans reload dur nécessaire — lu à chaque appel).

Pose du cookie (console, sur le domaine prod) :
```js
document.cookie = "suimini_data_layer=api; path=/; max-age=86400; samesite=lax";
// vérifier :
window.__suiminiDataLayer();      // → "api"
// rollback :
document.cookie = "suimini_data_layer=; path=/; max-age=0";
```

## 3. Incident RLS `tree_members` — résolu (rappel du fix)

- **Cause** : les policies `*_members_read` (0010) sous-requêtaient `tree_members`
  dans le filtre RLS combiné → `trees?select=*` / `persons?select=*` pouvaient **500**
  pendant une panne `tree_members`, **même pour le propriétaire**.
- **Fix C (migration `0017_resilient_members_read.sql`, APPLIQUÉE)** :
  1. Test propriétaire = **disjoint de premier niveau sans `tree_members`**
     (court-circuit).
  2. Test membre via **`is_accepted_member()`** — `SECURITY DEFINER` + `EXCEPTION
     WHEN OTHERS → false` (**fail-closed** : membres perdent la lecture, jamais
     d'erreur propagée).
  - **Validé sur Postgres jetable** : `tree_members` droppée → le propriétaire lit
    toujours `trees` (1) et `persons` (1).
- **`sharing.sql` restauré** : porte désormais la **version résiliente** (fonction +
  policies), donc restaurer l'accès membre post-incident = simplement ré-exécuter
  `sharing.sql` (= même contenu que 0017).
- **Fix B (déployé)** : `SyncFailedState` — un chargement échoué sans arbre affiche
  « Impossible de charger vos arbres — Réessayer » (rassure : données intactes côté
  serveur), **plus jamais** l'onboarding « Commencez votre arbre ».

⚠️ La **mitigation d'urgence** (DROP des 3 policies `*_members_read`) n'est **plus
nécessaire** : 0017 rend le DROP inutile. Si elle avait été laissée, `sharing.sql`
(déjà relancé) a recréé les policies.

| PR6 | Track mobile + prépa bascule | `docs/phase0-mobile-track.md` |

**Post-PR (résolution de l'échec de flip)** :
- Flag **runtime cookie-first** (`getDataLayer`, défaut `direct`) — l'échec initial
  venait de `NEXT_PUBLIC_*` **inliné au build** (un flag posé après coup ne change
  pas un bundle compilé). Corrigé (`4e1ef8b`).
- `getServerAuth` durci : cookie `setAll` réel (persiste le refresh de token) +
  `Authorization: Bearer` (mobile) (`4e1ef8b`).
- **SW v5** + sondes `window.__suiminiDataLayer()` / `/api/data/whoami` (`2262028`) —
  le v4 servait encore l'ancien code → le cookie n'était jamais lu.

**Design** : `docs/phase0-data-api-design.md`.

   frontière `DataClient`/`/api/data/*` rend le backend interchangeable). Pas avant —
   c'était la vraie dépendance derrière « migrer vers Railway ». Diagnostic complet à
   refaire à ce moment-là.
