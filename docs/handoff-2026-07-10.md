### 2.b — HORS PÉRIMÈTRE par conception (identité / accès / auth)

Directs **volontairement** — ce ne sont pas des données d'arbre, mais le bootstrap
d'identité/accès. Ne PAS confondre avec 2.a :

| Requête (host supabase) | Source | Raison |
|---|---|---|
| `/auth/v1/user`, `/auth/v1/token` | `useAuth` (`getSession`, `onAuthStateChange`) | Primitive GoTrue : établit/rafraîchit la session. **Ne peut pas** transiter par `/api/data/*` (circulaire). Exclusion permanente. |
| `rest/v1/profiles?...` | `useAuth:55`, `HomeGate:86`, `SettingsView:87,103` | Profil/rôle/locale self-only (RLS `id = auth.uid()`). Identité, pas données d'arbre. |
| **`rest/v1/tree_members?select=*&user_id=eq.<uid>&status=eq.accepted`** | **`fetchMyMemberships()` (`sharing.ts:191`) appelée par `useAuth:82` et `:108`** | **Liste les arbres partagés AVEC moi** (membre accepté) au bootstrap. Contrôle d'**accès**, pas de contenu. Fail-open : `error → []` (une panne tree_members ⇒ 0 arbre partagé, mais les arbres possédés chargent quand même via `/api/data/*`, résilient depuis RLS 0017). |
| `rest/v1/tenants`, autres `tree_members` | `useAdminData`, `sharing.ts` (ShareModal : `fetchMembers`, `inviteMember`) | Admin + gestion des membres, déclenchés par action, pas au chargement. |
| `wss://…/realtime/v1/websocket` | `SuiminiApp:345` (`postgres_changes`) | Abonnement temps réel — direct par conception (le realtime ne peut pas proxy par `/api/data`). Notifications, pas de chargement de données. |
| `rest/v1/*`, `storage/v1/*` **server-side** | routes `app/api/*`, `authz.ts`, `supabaseSync` appelé par les routes | **Serveur** — déjà du bon côté de la frontière. |

> ⚠️ **Correctif au trace du 2026-07-10 (matin)** : `tree_members` était classé « ne
> fire pas au chargement authentifié ». **Faux** : `fetchMyMemberships` (appelée par
> `useAuth`) le déclenche à chaque bootstrap de session. Ce n'est PAS un oubli du
> scope Phase 0 (le DataClient ne couvrait que le **contenu** d'arbre), mais un flux
> d'**identité/accès** du même bucket que `profiles`/`/auth/v1/user`. Candidat à une
> future passe « endpoints identité » seulement si on vise 0 requête navigateur→supabase.

## 3. Incident RLS `tree_members` — résolu (rappel du fix)

- **Cause** : les policies `*_members_read` (0010) sous-requêtaient `tree_members`
  dans le filtre RLS combiné → `trees?select=*` / `persons?select=*` pouvaient **500**
  pendant une panne `tree_members`, **même pour le propriétaire**.
- **Fix C (migration `0017_resilient_members_read.sql`, APPLIQUÉE)** :
  1. Test propriétaire = **disjoint de premier niveau sans `tree_members`**
     (court-circuit).
  2. Test membre via **`is_accepted_member()`** — `SECURITY DEFINER` + `EXCEPTION
     WHEN OTHERS → false` (**fail-closed** : membres perdent la lecture, jamais
     d'erreur propagée).
  - **Validé sur Postgres jetable** : `tree_members` droppée → le propriétaire lit
    toujours `trees` (1) et `persons` (1).
- **`sharing.sql` restauré** : porte désormais la **version résiliente** (fonction +
  policies), donc restaurer l'accès membre post-incident = simplement ré-exécuter
  `sharing.sql` (= même contenu que 0017).
- **Fix B (déployé)** : `SyncFailedState` — un chargement échoué sans arbre affiche
  « Impossible de charger vos arbres — Réessayer » (rassure : données intactes côté
  serveur), **plus jamais** l'onboarding « Commencez votre arbre ».

⚠️ La **mitigation d'urgence** (DROP des 3 policies `*_members_read`) n'est **plus
nécessaire** : 0017 rend le DROP inutile. Si elle avait été laissée, `sharing.sql`
(déjà relancé) a recréé les policies.
